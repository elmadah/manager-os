Projects Roadmap — Implementation Plan

manager-os • 2026-05-11 • Author: Claude • Status: Draft for review

TL;DR

Add a Roadmap tab to the Projects area that renders a Gantt-style timeline of all projects with their features nested underneath. Reuses existing projects.start_date / target_date and features.start_date / target_date — no schema changes. One new lightweight API endpoint, one new page component, two presentational components. Filters by status/health and by date range. Click a bar to open the project or feature.

1. Goals & Non-Goals

Goals

  • Single time-based view across all projects and their features.
  • Show schedule overlap, slip risk, and delivery clustering at a glance.
  • Drill in: click a project bar → /projects/:id; click a feature bar → project detail with feature focused.
  • Filter by project status, health, and a visible date window (quarter / half / year / custom).
  • Render entirely client-side after a single fetch; smooth for ~50 projects × ~10 features.

Non-Goals (YAGNI)

  • No drag-to-edit dates from the roadmap (edit stays in detail page modals).
  • No dependencies / blocked-by arrows between features.
  • No swim-lanes by team member (the existing /api/timeline already does that).
  • No baseline vs. actual comparison.
  • No print/export. (Browser print is fine if asked later.)

2.3 Empty & degraded states

No dates set

Projects/features without start_date or target_date are listed in a collapsed "Unscheduled" group above the timeline. From there a user can click into the project to add dates.

Out of range

Items that fall entirely outside the visible window are hidden but counted in a "+3 hidden" pill at the right edge with a button to expand the range.

3. Data Flow

One round trip. The page makes a single GET /api/projects/roadmap?start&end call. All re-grouping, layout math, and filter application happens in the client. Changing the visible range refetches; flipping status/health filters does not.

4.2 Why a new endpoint instead of reusing GET /api/projects?

  • GET /api/projects already returns nested features, but with no story stats per feature and no range filter — we'd over-fetch on every range change and still miss per-feature progress.
  • The existing GET /api/timeline groups by team member; reusing it would force client-side regrouping over a much larger payload.
  • A purpose-built endpoint keeps the SQL focused (two queries: projects-in-range + features-in-range with story aggregates) and the payload tight.

4.3 SQL sketch

-- Projects that overlap the window OR are unscheduled
SELECT p.*, ...story_stats...
FROM projects p
LEFT JOIN ( /* same story aggregate as GET /api/projects */ ) s ON s.project_id = p.id
WHERE p.start_date IS NULL OR p.target_date IS NULL
   OR (p.target_date >= ? AND p.start_date <= ?);

-- Features for those projects, with per-feature story aggregate
SELECT f.*, COUNT(st.id) AS total, SUM(...) AS completed, ...
FROM features f
LEFT JOIN stories st ON st.feature_id = f.id
WHERE f.project_id IN (...)
GROUP BY f.id;

5.1 Bar position math (in lib/roadmap.js)

function computeBarPosition({ start, target, rangeStart, rangeEnd }) {
  const r0 = new Date(rangeStart).getTime();
  const r1 = new Date(rangeEnd).getTime();
  const s  = Math.max(new Date(start).getTime(),  r0);
  const e  = Math.min(new Date(target).getTime(), r1);
  if (e <= s) return null;          // entirely outside window
  const span = r1 - r0;
  return {
    leftPct:  ((s - r0) / span) * 100,
    widthPct: ((e - s) / span) * 100,
    clippedStart: s > new Date(start).getTime(),
    clippedEnd:   e < new Date(target).getTime(),
  };
}

Clipped flag drives a fade-out edge on the bar so users see "extends beyond view."

5.2 Feature health derivation (client-side)

Features don't have a stored health field. Derive from progress + dates:

  • Off track (red): status not complete AND target_date < today.
  • At risk (amber): completion_ratio < expected_ratio - 0.2, where expected_ratio is time elapsed in [start, target].
  • On track (green): everything else (including features with no stories yet).

6. URL & State Shape

Filter state lives in the URL so views are shareable and survive reloads:

/projects/roadmap?start=2026-04-01&end=2027-03-31&zoom=quarter&status=active,planning&health=green,yellow,red

Range presets: 3m, 6m, 12m (default, snapped to current quarter start), custom. zoom is month or quarter — the latter shows the Q-band header above months and is the default for 12m. Stored as concrete dates so the URL is stable when shared.

7. Implementation Steps

  1. Server: add GET /api/projects/roadmap in server/routes/projects.js above the /:id route. Two queries (projects + features) with one shared story-aggregate subquery; assemble in JS.
  2. Routing: in App.jsx change /projects to redirect to /projects/board; add /projects/board (Kanban) and /projects/roadmap (new page). Both share the same ProjectsPage shell that renders the tab bar + active route's <Outlet/>.
  3. Helpers: write lib/roadmap.js with computeBarPosition, monthTicks, healthOfFeature. No tests in repo, but functions are pure so a small console.assert-style sanity block is fine to keep until verified by hand.
  4. Presentation: build RoadmapGantt with CSS grid (220px sidebar + 1fr track) and absolute-positioned bars. No external Gantt lib — keeps bundle small and matches the codebase's "thin deps" preference.
  5. Toolbar: range presets, prev/next (shift window by half its size), Today button, status & health multi-select.
  6. Container: ProjectsRoadmapPage reads URL params, fetches once per range change, memoises filtered projects, passes to children.
  7. Unscheduled section: collapsible card listing projects/features with missing dates; each item links to the detail page.
  8. Manual QA: verify with the seed dataset (cd server && npm run seed) and a hand-crafted "project crossing the window edge" case.

8. Edge Cases

  • Feature ends before project starts (data entry mistake): show the feature bar where its own dates say, regardless of project span.
  • Project has no features: still show the project row with just the project span bar.
  • Feature outside project span: render normally; do not auto-clamp — surface the inconsistency rather than hide it.
  • start_date = target_date: render a 4px-wide milestone diamond instead of a bar.
  • Very long project (> window): bar visually extends to both edges with a faded gradient.
  • Time zones: dates in DB are YYYY-MM-DD strings; parse as local midnight and avoid new Date('2026-05-11') which is UTC — use new Date(y, m-1, d).

9. Performance Notes

  • Expected payload: ~30 projects × ~8 features ≈ 30 KB JSON. No virtualisation needed.
  • Memoise filtered + positioned rows with useMemo keyed on [data, filters, range].
  • Server query is two indexable selects; story-aggregate already exists in GET /api/projects with acceptable latency.

10. Risks & Open Questions

Risks

  • Health heuristic for features may feel arbitrary — keep the formula visible in the tooltip ("expected 60% by today, at 38%").
  • If many features lack dates, the unscheduled list could dominate. Mitigation: cap to first 10 with "show more."

Open questions

  • Should the roadmap be the default Projects tab (vs. the current board)? Recommend: keep board as default to avoid surprising existing users; remember last choice in localStorage.
  • Quarter-band header is now the default for 12m. For 3m/6m windows, hide the Q-band and show only months — confirm that's the right default.
  • Year boundary in the 12m view: should we render a stronger vertical rule between Dec and Jan (currently a quarter divider) and a small "2027" label? Cheap to add.
  • Color source: use project.color for the project span bar and a status-derived color for feature bars — confirm this is the desired hierarchy.

11. Rollout

  1. Single PR. No migration, no env changes.
  2. Manual smoke test with seeded DB, then with a backup of the user's real DB.
